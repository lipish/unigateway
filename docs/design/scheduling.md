# UniGateway 调度与排队演进路线图 (Scheduling & Queueing Optimization Roadmap)

当前网关侧已实现的基础排队与并发行为见 [`queue.md`](./queue.md)；本文档描述其后的演进方向。

## 设计哲学与定位

UniGateway 作为**基础底层服务网关**（旨在为类似 OpenRouter 的上层大规模应用提供弹药库），在设计调度与队列优化时，我们秉承以下核心哲学：

**“吸收底层的流量调度与弹防能力，坚决剥离上层的用户态业务逻辑。”**

底座引擎的任务是：把流量平顺地送进下游 Provider、榨干一切通道并发资源、且绝不把 Provider 通道打爆；至于计费、具体的 C 端用户限流规则、或者“用户还有没有钱用 o1 模型”，这些应全盘交给上层应用。

因此，基于我们近期对孪生项目经验的回顾与提炼，我们将 UniGateway 的排队与并发改造分为以下梯次落地：

## Phase 1: 活下来与稳步整流（最强防御基座）

**核心目标：防止突发流量冲垮基座，消除不必要的 Fail-fast 429 报错，让网关成为真正的“减震弹簧”。**

### 1。 AIMD 下游自适应并发控制 (Adaptive Concurrency)
*   **痛点**：目前在 Route 上配置静态并发上限 (`concurrency_limit`) 既浪费资源又易遭 429 反噬。
*   **落地方案**：为底层每个 Provider 实例引入 TCP 慢启动和拥塞避免机制。当下游返回成功时动态上调并发窗口；当遇到 429 或 Timeout 瞬间乘法减小窗口（如斩半）并冷却。通过这一步，彻底免除“探底测试”，让网关时刻自动寻找最佳并发压强。

### 2。 QPS 突发转短暂睡眠 (Burst to Sleep)
*   **痛点**：当前达到 QPS 或 Token 额度上限时，网关直接对外拒绝，造成调用方重试甚至雪崩。
*   **落地方案**：借鉴 Token Bucket 借贷未来的思想，当瞬间并发超过令牌数时，不再直接 429，而是估算需要等待的时间（在设置的容忍时长内），在请求执行前短促地 `tokio::sleep`。完美削平尖峰方波。

### 3。 SRE 级的极限观测 (SRE Observability)
*   **痛点**：当发生请求阻塞时，往往不知道是网关出问题了还是 Provider 慢了。
*   **落地方案**：摒弃面向上层用户的复杂图表，专注系统级的内部观测点。重点暴露各个 API Key 下当前的 `in_flight`（执行中）、本地 `sleepers_count`（等待额度中）以及 Provider 的真实被击退次数，确保白盒化维护。

## Phase 2: 质量评估与内部公平性（QoS 与防霸占）

**核心目标：在第一步彻底保障了系统存活率并跑出稳定样本数据后，进一步构建系统的评估能力和多租户公平。**

### 1。 Per-Model 独立延迟追踪与 ETA 预估 (EMA/ETA)
*   **痛点**：生成型模型 (Haiku) 和长推理模型 (o1) 延迟相差几十倍，混在一个队列里估算时间是彻底无效的。
*   **落地方案**：按照 Model 分离 Exponential Moving Average (EMA) 统计。利用实际样本平滑计算各个模型的处理耗时。该预估时间的价值不在于向末端用户展示，而是用于底座的**内部过载准入拦截 (Fail-fast before routing)**，或者通过内部 Header 暴露给 OpenRouter 作为异地多活的切流参考。

### 2。 基建级多租户公平防霸占 (Tenant Depth Safety Nets)
*   **痛点**：即便业务限流在上层做，如果一个业务线的压测脚本失控，网关底座依然会被瞬间塞满，导致整个公司瘫痪。
*   **落地方案**：设置基于底层队列全局深度的“租户隔离比”。单个租户（API Key/Service）不论多么狂暴，其排队深度总和不得占据全局协程或者排队坑位的一个上限百分比，保证基础防御体系永远不会只服务一个人。

---

## Phase 3: 技术视角的模型扩展（可选能力）

### 技术导向的模型约束 (Technical Model Awareness)
*   并不是要把 C 端配额引擎塞入基座，但保留对于高危、特异模型的“技术整形”扩展能力。
*   比如对于特定高价值模型给予纯技术架构上的硬并发阻断，或对响应奇慢的长链路模型另开 Fast/Slow 线车道池以防资源饥饿。此能力只作为基础组件的可选扩展点默认关闭。

---

> **一句话总结路线**：先用 AIMD 与短时缓冲保证网关死不掉且护住下游；再去用细粒度 EMA 指标与租户队列配额保证 QoS 公平与透明化。
